Project Summary

ToothTime is a dental clinic management system that allows dental clinics to manage their patients, appointments. The objective was to build a system that faciltates the management of the dental clinic and better handle late cancellations and no-shows.

Objectives

  • Improve the efficiency of the dental clinic
  • Handle late cancellations and no-shows
  • Reduce lost revenue

Goals

  • Create a responsive and user-friendly interface
  • Implement a robust backend system
  • Handle concurrent requests
  • Ensure data security and privacy

Development Approach

Architecture: The Modular Monolith

Instead of a premature "Microservices" mesh, I chose a Modular Monolith using Spring Modulith.This approach allows for a single codebase with a clear separation of concerns, making it easier to maintain and scale. The modular monolith is a great way to start a project, it allows for a quick and easy development process, and it is a great way to learn the basics of the Spring framework.

Implementation:
  • Hard Boundaries: The Patient module cannot import the Appointment module.
  • Event-Driven Communication: When a patient is registered, an internal PatientCreatedEvent is published. The Notification module listens to this asynchronously to send emails. This decouples the transaction time from 3rd party API calls (SendGrid/Twilio).

The Tech Stack

  • Backend: Java 21, Spring Boot 3.3, Spring Data JPA.
  • Database: PostgreSQL (Schema-per-Module strategy to ensure data isolation).
  • Frontend: Next.js 14 (React), TypeScript, Tailwind CSS (shadcn/ui).
  • DevOps: Docker & Kubernetes.

Outcome

For the Clinic (Business Value)

  • Zero Overlaps: The database-level CHECK constraints and transactional locking logic successfully prevent 100% of double-booking attempts.
  • Optimistic UI: The frontend feels "instant." It assumes success and only rolls back on the rare 409 Conflict, making the app feel responsive even on mobile networks.
  • Reliability: The system handles third-party failures (e.g., Email service down) gracefully because notification logic is async—appointments are saved even if the email fails.

For Me (Learning and Growth)

  • Mastered Domain-Driven Design (DDD): I learned how to separate the "Core Domain" (Business Rules) from the "Infrastructure" (Database/Web) using Hexagonal Architecture.
  • PostgreSQL Deep Dive: I moved beyond simple CRUD to using advanced features like schema-per-module and JSONB indexing.
  • Testing: By using Testcontainers, I stopped writing brittle mocks for my database. My tests now run against a real Postgres instance in Docker, giving more reliable tests.

Project Summary

This project is a full end-to-end DevOps pipeline built around a containerized Spring Boot application with a PostgreSQL database, deployed on Kubernetes and supported by modern CI/CD, environment automation, and advanced deployment strategies.

Objectives

  • Learn how to build a full end-to-end DevOps pipeline
  • Learn how to deploy multiple services on Kubernetes
  • Learn how to use Jenkins for CI/CD

Goals

  • Build a functional DevOps pipeline in a production-like environment
  • Successfully deploy the application on Kubernetes using blue/green deployment strategy

Features

Pipeline

The system uses Jenkins to run continuous integration (tests, build, image creation) and continuous delivery to three permanent environments—dev, staging, and prod—with environment-specific configurations supplied through Helm. Staging and production use a managed cloud PostgreSQL database, while dev and short-lived environments use in-cluster Postgres for simplicity.

On-Demand Environments

every pull request automatically creates its own isolated Kubernetes namespace with its own application instance and database. When the PR is closed, the entire environment is destroyed. This demonstrates automated environment provisioning, cleanup, and preview deployments.

Release Strategies

  • Blue/green deployments allow a new version to be deployed to an idle environment, verified, and then switched live with a traffic flip, providing instant rollback.
  • Canary deployments (via Argo Rollouts) progressively shift traffic to safely expose users to new releases.

Project Summary

A cross-platform application designed to assist users in managing financial goals through an AI-powered chatbot. Leveraging OpenAI's GPT API, the app makes financial planning conversational, accessible, and engaging. This project was delivered within a three-month deadline for a real-world client—a professional financial coach. The solution serves as a preliminary tool for her company, enabling clients to organize their financial goals autonomously before engaging in 1-on-1 coaching.

Objectives

  • Simplify financial goal tracking
  • Provide AI-powered insights
  • Enable cross-platform access

Goals

  • Build intuitive conversational UI
  • Integrate GPT for natural language
  • Ensure data security
  • Support web and mobile platforms

Technical Approach

Frontend

Angular for the web application and React Native for mobile, ensuring a consistent user experience across platforms with shared business logic.

Backend

Python backend with Flask API, integrated with OpenAI's GPT for natural language processing and financial advice generation.

AI Integration

Custom prompt engineering to provide context-aware financial guidance, goal tracking, and personalized recommendations.
